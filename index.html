Offline viewing and editing spatial data for Forest Fire Defense

Feedback on the GeoPoppy solution

Michaël Douchin - 3liz

# Context ## Stakeholders, data, softwares
## Stakeholders * **Valabre**: french public institution * **Help and support actors**: prevent and fight natural risks (fire, flood, etc.) * **Actions**: test new technologies, research, trainings, share services, tools and data. * **Partners**: cooperation between several organizations in the French Mediterranean areas. ![Valabre](media/Logo-Valabre_400.png)
 
## History: consolidate data between partners * **Data**: fire fighting infrastructure: water tankers, tracks, watchtowers + standardized grids + base layers * **Harmonization and centralization** between actors: * list of **layers** to share (name, metadata, ontology) * agree on data structure (fields, constraints, etc.) * create common glossary * aggreement on data styling (symbols, colors, etc.) * **Spatial extent**: French Mediterranean areas (125 965 km2) * **Vectors** 200 Mo / **Rasters** 30Go
 
 
## Softwares already in use * **QGIS** used by *Valabre* workers, but some actors also use Mapinfo/ESRI/Geoconcept * **PostgreSQL** to store and process data: * Dedicated server (hosted by 3liz) * **Lizmap Web Client / QGIS Server** to publish web maps from QGIS ![Logo QGIS](media/qgis-icon.jpg) ![Logo PostgreSQL](media/logo-postgresql.jpg) ![Logo Lizmap](media/logo_lizmap.jpg)
## Lizmap Web Client * **QGIS projects published as full featured web maps** * Lizmap plugin to configure the map * Respect QGIS project features & properties: styling, layer tree, attribute table, form editing, printing via composers, relations * **Authentication and group rights**: restrict map access, editing tools, data export, etc. ![Logo Lizmap](media/logo_lizmap.jpg)
 
 
 
 
 
 
 
## PostgreSQL Valabre database * **One schema** per actor (administrative area) + schema dedicated for admins or auditing. * **Rights managements** via GRANT on schemas and tables * **Procedures, triggers and Python services**: * default values, * history audit, * send emails to admins/users depending on data modification/validation, * copy data accross local schemas and global schema ![Logo PostgreSQL](media/logo-postgresql.jpg)
# Project ## See and edit data offline ## mainly in the field
## Requirements and constraints * **GIS features**: multiple layers, data editing, baselayers (aerial photos, dem), attribute table, etc. * Must be **based on QGIS projects** to simplify the workflow & respect harmonized styling * Use **PostgreSQL in the field**: constraints, triggers, views (buffers, etc.) * **No internet connection** or connection **not stable or expensive** -> work offline * **Heavy datasets**
## Features requirements * **Insert/update/delete** features * **Rich forms needed**: lists, checboxes, images, date fields, etc. * Keep **Lizmap features**: * authentication and rights management -> login needed and saved in data * tools: attribute tables, popups, filters, relations * **Synchronization** * between the **offline database copies** * and the **central server database**. * No conflit resolution needed.
## Studied solutions / needs * Android tablets / Windows tablets * Simple field data collection tool / QGIS based mobile solution * For the latter: QField / Input / GeoPoppy Valabre chose to test **GeoPoppy** ![Logo Lizmap](media/logo_geopoppy.png)
# Geopoppy ## Description / testing phase
## Geopoppy ### A spatial server in a box * **Hardware**: RaspberryPi / Battery / Tablet or smartphone or laptop with a web browser (Firefox) * **Software**: Lizmap Web Client / QGIS Server / PostgreSQL / FTP Server. * **Main conceptor**: **Julien Ancelin** with some help by 3liz * Based on **Docker** * **Already used** by researchers in France, Ivory Coast, Madagascar, etc.
 
 
 
## Geopoppy cons * **Additionnal devices**: *Do-It-Yourself look&feel* * Tied to the **features of web browser**: geolocation API, no access to files, etc. * Lizmap interface is responsive but **not dedicated to mobile**: some improvements needed * Cannot connect to 5G and Geopoppy WIFI **at the same time**: **Android OS restriction**
## Geopoppy pros * **OpenSource** and cheap * **Work offline** * **No new software**: QGIS Server / Lizmap / PostgreSQL * **Same worklow** to publish the web maps and data to the central server * **PostgreSQL powered** : views, constraints, etc. * **QGIS Server powered**: all raster/vector formats (no conversion needed) * **Lizmap powered**: authentication and rigths, rich forms from QGIS, styles, etc.
# Synchronization ## concepts and tools
## Database synchronization key ideas * **package** from server/ **deploy** to device / **synchronize** data * **No conflicts management**: last action wins * **UUID field** in every table: *c0558823-d20a-40c8-903c-0d8c4f4dad7d* * **Clone partial database** in each Geopoppy ( unique ID per device) * On update, **only changed fields** data is synchronized
## Database synchronization tools * Based on **PostgreSQL triggers/functions**: *audit.sql* with additionnal sync procedures * **Log every actions** with many information : * initial raw data before change & changed fields * user and app, timestamp, * uid of the source device, etc. * **Centralize all audit** log in the main server * **Sync process**: * Get central server modifications * Then send local modifications

# Conclusion ## Feedback, roadmap, ideas
## Feedback * **It works !** * in the field, * but also as an offline **portable full GIS server** (WMS, WFS, WPS, PostgreSQL, etc.) * Need to **improve the workflow** : simplify processing algs * Improve **PyQGIS WPS Server** and Lizmap client * Complex QGIS projects with many layers are **slow to render** * **Lizmap responsive web interface** can be improved
## The future ### GeoPoppy on Android ! * All the stack **inside your Android device** * No additionnal hardware devices needed: **only your tablet** * PostgreSQL running on Android: https://github.com/CypherpunkArmory/UserLAnd/issues/256 * Much **better performances** * **Almost production ready**: some work needed to package this solution.
